# Design: cmdk palette v1.3.0 — Commands (stateless global verbs)

Fifth installment of the command palette. v1.0 shipped the core palette
(smart-search + people + places + tags + page navigation). v1.1 (#362) added
albums and spaces sections. v1.2 (#365) added `@ # / >` prefix scoping.
v1.2.1 follow-ups (#367) fixed people thumbnails and added a placeholder
hint.

v1.3.0 turns the `>` scope from "pages-only" into "do-or-go" by adding real
verbs — the half-scaffolded `ACTIONS` branch already in `navigation-items.ts`
(currently a single `nav:theme` item) grows into a first-class `CommandItem`
registry. Seven stateless global verbs ship in v1.3.0; larger scopes
(settings toggles, admin verbs, context-aware commands, selection verbs) are
sketched as v1.3.1 → v1.5 for easy pickup.

## Non-goals (v1.3.0)

- **Context-aware commands** (route/params/selection-driven availability) —
  deferred to v1.4. The CommandContext infra has stale-context failure modes
  that deserve their own design round.
- **Selection verbs** (Add to album, Favorite, Archive on selected assets) —
  deferred to v1.5. Requires plumbing `selectedAssetIds` through the palette
  and multi-step "pick then execute" commands (album picker, etc).
- **Admin verbs** (run job, pause queue, clear failed jobs) — deferred to
  v1.3.1. Type has `adminOnly?: boolean` reserved but no item uses it in
  v1.3.0.
- **Settings toggles** (map-tile theme, show-in-timeline default,
  auto-classification on/off) — deferred to v1.3.1.
- **Destructive-verb confirmation UX** (inline confirm row, typed confirm,
  double-Enter) — deferred to v1.4, introduced alongside context-aware verbs.
  v1.3.0 ships sign-out as a direct action matching every other palette row.
- **Frecency ranking** across commands + other sections — deferred.
- **Per-command keybindings** (Ctrl+Shift+U for Upload etc.) — deferred.
  Needs a keybind-registry system we don't have.
- **First-run callout / onboarding tooltip** — premature.
- **Bridging `@immich/ui`'s `CommandPaletteDefaultProvider`** — rejected.
  The registry lives in a different lifecycle than our palette, and
  extending our `@immich/ui` patch (currently 23 lines) to expose it
  balloons the fork surface. Our in-repo dispatcher already handles the one
  existing action (`nav:theme`); extending it is the minimum-diff path.

## Command inventory (v1.3.0)

7 stateless global verbs. Every handler is a zero-arg call to a primitive
that already exists in the codebase — no new business logic.

| #   | Command               | Label key                    | Icon                     | Handler                                                                         |
| --- | --------------------- | ---------------------------- | ------------------------ | ------------------------------------------------------------------------------- |
| 1   | Upload files          | `upload`                     | `mdiCloudUploadOutline`  | `openFileUploadDialog()`                                                        |
| 2   | Create album          | `new_album`                  | `mdiPlaylistPlus`        | `createAlbumAndRedirect()`                                                      |
| 3   | Create shared space   | `create_space` _(new)_       | `mdiAccountMultiplePlus` | `modalManager.show(SpaceCreateModal, {})`                                       |
| 4   | Sign out              | `sign_out`                   | `mdiLogoutVariant`       | `signingOutToast(); return authManager.logout()`                                |
| 5   | Keyboard shortcuts    | `keyboard_shortcuts`         | `mdiKeyboardOutline`     | `modalManager.show(ShortcutsModal, {})`                                         |

## Architecture

### New registry module

`web/src/lib/managers/command-items.ts`:

```ts
export interface CommandItem {
  id: `cmd:${string}`;
  labelKey: string;
  descriptionKey: string;
  icon: string;
  handler: () => void | Promise<unknown>;
  /** Reserved for v1.3.1 admin verbs. Not used by any v1.3.0 item. */
  adminOnly?: boolean;
  /** Reserved for future feature-flag gating. Not used in v1.3.0. */
  featureFlag?: keyof ServerFeaturesDto;
}

export const COMMAND_ITEMS: readonly CommandItem[] = [
  /* 7 items */
];

export function isAlmostExactCommandMatch(query: string, label: string): boolean;
```

`isAlmostExactCommandMatch` is the same algorithm as `isAlmostExactNavMatch`
today. Both call a shared helper `isAlmostExactWordMatch` extracted into
`navigation-items.ts` (or a new `cmdk-match.ts` — implementation detail).

### Removal: `nav:theme` → `cmd:theme`

`navigation-items.ts`:

- Delete the `ACTIONS` array (single item, `nav:theme`).
- `NAVIGATION_ITEMS` becomes `SYSTEM_SETTINGS + ADMIN_PAGES + USER_PAGES` only.
- The `NavigationItem.category` field keeps its existing values
  (`'systemSettings' | 'admin' | 'userPages'`) — `'actions'` is dropped.
  Still used to bucket pages under system-settings / admin / user-pages
  section headers.

`global-search-manager.svelte.ts`:

- Delete the `if (n.category === 'actions')` branch in `activate('nav')` at
  L1026-1037. The `activate('nav')` path now unconditionally falls through
  to the `goto(n.route)` + RECENT write path, because every remaining
  `NavigationItem` has a non-empty `route`.

### New section and `kind: 'command'`

Add `sections.commands` alongside the existing seven (`photos`, `people`,
`places`, `tags`, `albums`, `spaces`, `navigation`). **Do not** rename
`sections.navigation` — the rename touched ~10 call sites for zero
behavioral gain, and the reviewer flagged it as churn. The UI section
header for commands is a new `cmdk_section_commands` key
("Commands"); the existing `cmdk_section_navigation` key keeps rendering
pages as "Go to…".

#### Type changes (enumerated in one place)

- `Sections` (L60) — add `commands: ProviderStatus<CommandItem>`.
- `EntitySectionKey` (L104) — **stays as-is**. Commands, like navigation,
  are `ProviderStatus<CommandItem>` (not `ProviderStatus<EntityItem>`) and
  flow through a synchronous provider path; they must be excluded from
  `EntitySectionKey` for the same generic-assignment reason navigation is.
- `ActiveItem` union (L75-82) — add `{ kind: 'command'; data: CommandItem }`.
- `activate` kind union (L974) — add `'command'`.
- `ENTITY_KEYS_BY_SCOPE` (L105) — **no change**. Map is entity-only; the
  navigation/commands path is orthogonal to the entity dispatch lifecycle.
- `NavigationItem.category` — the literal `'actions'` drops out of the
  union (becomes `'systemSettings' | 'admin' | 'userPages'`).

#### Other touchpoints

- `sections` initializer (L~202) — add `commands: idle`.
- `resetSections` (L~745) — add `commands: idle` reset.
- `sectionForKind` switch (L~882) — new case `'command' → sections.commands`.
- `kindToSection` map (L~920) — `'command' → 'commands'`.
- `RECONCILE_ORDER_BY_SCOPE` (L123; **not** `DISPLAY_ORDER_BY_SCOPE` — that
  constant does not exist): prepend `'commands'` to the `'all'` and `'nav'`
  arrays. **Preserve the existing tail verbatim** — the comment at L121-122
  documents that this map doubles as render order, and reordering
  photos/albums/spaces silently regresses unscoped search UX.
  - `'all'`: `['commands', 'photos', 'albums', 'spaces', 'people', 'places', 'tags', 'navigation']`
  - `'nav'`: `['commands', 'navigation']`
  - Rationale for commands-first under `'all'`: a user typing `upload`
    unscoped most likely wants the verb (open a picker) over a match on an
    existing photo filename. Commands-first matches the v1.0 "nav is last"
    precedent inverted for commands (higher intent).
- `activeItemFromRecent` switch (L~800) — no new branch (commands skip
  RECENT), but add a guard comment noting why.

### Provider (two registrations sharing a helper)

The existing `providers: Record<keyof Sections, Provider>` invariant (one
provider per section key) is preserved. We register **two** providers —
`providers.navigation` and `providers.commands` — each calling a shared
helper `filterNavAndCommands(payload, scope)` that does the fuzzy-match
once and returns both filtered arrays; each provider picks its own half.
This keeps the 1:1 provider/section map intact and avoids the
"one provider writes two sections" wrinkle the earlier draft implied.

Behavior matrix:

| `scope` | `payload`        | Commands section                                                                             | Pages section                                     |
| ------- | ---------------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| `'nav'` | empty (bare `>`) | all 7 commands alphabetical by translated label                                              | all pages alphabetical (current behavior)         |
| `'nav'` | non-empty        | fuzzy-match payload across commands, top 5                                                   | fuzzy-match payload across pages, top 5 (current) |
| `'all'` | any              | fuzzy-match commands, top 5; also feed `isAlmostExactCommandMatch` into top-result promotion | unchanged                                         |

Commands' top-result promotion under `'all'`: if the query word prefix-matches
a command label, the command takes the top-result slot (ahead of photos /
people / etc.), same mechanism as pages use today via
`isAlmostExactNavMatch`. **Tie-break**: if the same query matches both a
command and a page (e.g. hypothetically "Upload files" command and an
"Upload" page), command wins. Enforced by ordering — commands are
checked first in the promotion path.

### activate('command')

Single-flight guard to prevent rapid-Enter double-fire (file picker ×2,
modal ×2). Borrowed from v1.1's `activationInFlight` pattern:

```ts
private commandInFlight = new Set<string>();

activate(kind: 'photo' | 'person' | 'place' | 'tag' | 'nav' | 'command', item: unknown) {
  // ... existing branches ...
  case 'command': {
    const cmd = item as CommandItem;
    if (this.commandInFlight.has(cmd.id)) return;
    this.commandInFlight.add(cmd.id);
    this.close();
    queueMicrotask(() => {
      void cmd.handler()
        .then(() => undefined)
        .catch((error) => logger.error('[cmdk] command handler failed', { id: cmd.id, error }))
        .finally(() => this.commandInFlight.delete(cmd.id));
    });
    return;  // no RECENT write
  }
}
```

**Close-then-handle order** via `queueMicrotask`:

- Palette closes synchronously on Enter (current behavior for nav).
- Handler runs on the next microtask — after the palette unmount commits —
  so modals (`SpaceCreateModal`, `ShortcutsModal`) mount cleanly into a
  cleared stack.
- Theme toggle still feels instant (microtask ≈ 0ms).
- Sign out: palette closes, toast shows "Signing out…", redirect happens.
- Errors surface via each handler's own toast/logger, not via the palette.

### Rendering

- New `CommandRow.svelte` modeled on `NavRow.svelte`. Icon + translated
  label + translated description. No secondary metadata (no count, no
  relative time). Same paddings and focus/hover treatment as NavRow so the
  two sections look uniform under `>`.
- Section header: new i18n key `cmdk_section_commands`
  ("Commands"). Renders above the existing `cmdk_section_navigation`
  header when both have items.
- Preview pane: empty for commands (falls through to the same
  "Select a result to preview" empty-state as nav items).
- **Section container**: new `GlobalSearchCommandsSection.svelte` modeled
  on `GlobalSearchNavigationSections.svelte`. Wired into
  `web/src/lib/components/global-search/global-search.svelte` at the two
  existing navigation mount points (L493 for `'all'` scope, L558 for
  `'nav'` scope) — the Commands section renders immediately **before** its
  paired `<GlobalSearchNavigationSections>` at each site. This matches the
  `RECONCILE_ORDER_BY_SCOPE` ordering and keeps the render/focus order in
  sync with the cursor-reconcile order.
- **Top-result de-duplication**: when a command is promoted into the
  top-result slot under `'all'` via `isAlmostExactCommandMatch`, it is
  **omitted** from the Commands section below to avoid rendering the same
  row twice. The commands-provider filter path subtracts any id already
  claimed by the top-result slot before slicing to `topN`. Symmetrical to
  the existing nav promotion behavior.

## Error handling

| Failure mode                                             | Behavior                                                                                                                                                                                                                                                                                                                |
| -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Handler throws synchronously                             | `.catch` in the microtask chain logs via `logger.error`. No user bubble.                                                                                                                                                                                                                                                |
| Handler returns rejected promise                         | Same `.catch`.                                                                                                                                                                                                                                                                                                          |
| `openFileUploadDialog` user cancels native picker        | Native picker resolves with no files; `fileUploadHandler` returns a no-op. No error.                                                                                                                                                                                                                                    |
| `authManager.logout()` network fail                      | `authManager.logout()` redirects to `/auth/login` regardless (see `auth-manager.svelte.ts` L31-49). Toast shows "Signing out…" during the in-flight call so the user isn't staring at a closed palette in silence.                                                                                                      |
| `modalManager.show()` rejects (modal fails to construct) | `.catch` logs; palette is already closed, user sees no UI change. Not worth a toast — indicates a bug, surfaces in logs.                                                                                                                                                                                                |
| `clearEntries()` invoked while palette still open        | Allowed. `clearEntries` (actual export in `cmdk-recent.ts:97`) writes `[]` to localStorage synchronously. The RECENT section re-derives on next palette open; no live update needed in v1.3.0 (the row self-disappears after palette close).                                                                            |
| Palette closed mid-handler (Escape right after Enter)    | `close()` aborts `closeSignal`; command handlers are fire-and-forget, do not subscribe to `closeSignal`. The already-scheduled microtask runs to completion (upload picker mounts, modal mounts, logout redirects). Accepted: commands complete once activated, cancellation mid-activation is not supported in v1.3.0. |
| Logged-out user activates `cmd:clear_recents`            | `currentStorageKey()` returns null → `clearEntries` is a no-op. Harmless. The command stays listed even for logged-out users (they shouldn't see the palette at all in that state; covered by auth redirect upstream).                                                                                                  |
| `cmd:signout` racing another pending command             | Logout redirect interrupts. The other handler's microtask may have fired already; its side-effects (modal mount, file picker) are orphaned by the navigation. Accepted — edge case requires double-fast-Enter on two different commands.                                                                                |
| i18n key missing in non-en locale                        | `t('<missing-key>')` renders the key literal (standard codebase behavior). No crash. Documented, not tested.                                                                                                                                                                                                            |

Handler contracts: every v1.3.0 handler is a known-safe primitive that
either returns void/Promise<void> or navigates/redirects. None write state
the palette consumes directly.

## Recents coherence

Commands skip RECENT writes — matches existing behavior for `nav:theme`
(see comment at `global-search-manager.svelte.ts:1031`: "persist a broken
recent"). Commands are verbs, not places to return to.

**No migration needed.** `isValidRecentEntry` at
`global-search-manager.svelte.ts:131-168` already requires `kind ===
'navigate'` entries to have `route.length > 0` (L156-157); the old
`nav:theme` item has `route: ''` (navigation-items.ts:252). Any stale
entry is filtered by `activateRecent` at L1059 on read. A belt-and-braces
`rawRead`-level filter was considered and rejected as dead code. One spec
asserts this behavior (see Testing).

## Logout-while-palette-open

`authManager.logout()` is async (network request before redirect, L31-49
of auth-manager). Without mitigation, user sees "palette closed, nothing
happening" for up to a few seconds on slow networks.

Mitigation: the `cmd:signout` handler fires an info toast
(`t('signing_out')` → "Signing out…") on the first line, before awaiting
`authManager.logout()`. Toast dismisses automatically on navigation (via
existing SvelteKit navigation teardown) once the redirect lands.

Trade-off: adds one string. Accepted.

## Alternatives considered

- **Bridge `@immich/ui`'s `CommandPaletteDefaultProvider`**: rejected. The
  provider registry lives in a different lifecycle; extending our
  `@immich/ui` patch (23 lines today) to expose it balloons the fork
  surface. Our in-repo dispatcher already handles the one existing action,
  so extending is minimum-diff.
- **`!` prefix for commands, `>` pages-only**: rejected. `>` = "do or
  go" matches Linear / Raycast mental models; splitting prefixes adds a
  second thing to remember.
- **Single combined "Pages & commands" section under `>`**: rejected.
  Verbs next to destinations blur intent; the two-section split gives
  commands room to grow (v1.3.1, v1.4) without drowning in pages.
- **Context-aware verbs in v1.3.0**: deferred. Stale-context failure
  modes deserve their own design round; ship the stateless slice first.
- **Rename `sections.navigation` → `sections.pages`**: rejected per
  code review. ~10-site churn for zero behavior; adding
  `sections.commands` is the minimum diff.
- **Confirm UX for sign-out in v1.3.0**: rejected. Matches every other
  direct-action row. Inline-confirm infra is v1.4.

## Future sub-versions (documented for easy pickup)

### v1.3.1 — Settings toggles & admin globals

Extend `COMMAND_ITEMS` using the already-present `adminOnly` and
`featureFlag` fields. No new infra.

Candidate items:

- Toggle map-tile theme (dark / light) — writes to user preferences
- Toggle show-in-timeline default — user pref
- Toggle auto-classification enabled (admin only) — system config
- Run job: Thumbnail generation (admin only) — `POST /api/jobs/:name/run`
- Run job: Metadata extraction (admin only)
- Run job: Smart-search (admin only)
- Pause all queues (admin only)
- Clear failed jobs (admin only)

Provider gates `adminOnly` via the user's `isAdmin` flag on the auth store
and `featureFlag` via `featureFlagsManager.valueOrUndefined`. Both fields
already on `CommandItem`.

### v1.4 — Context-aware commands

New module `web/src/lib/managers/command-context-manager.svelte.ts`
exposing reactive `{ route, params, selection }` derived from SvelteKit's
`page` store and a selection subscribe. Updates on navigation and
selection change.

`CommandItem` gains:

```ts
isAvailable?: (ctx: CommandContext) => boolean;
handler: (ctx: CommandContext) => void | Promise<unknown>;
```

Target verb set (album page):

- Rename this album
- Share this album
- Delete this album (destructive — uses new inline confirm UX)
- Add to this album (opens asset picker modal — still single-step,
  multi-step deferred to v1.5)

Inline confirm UX introduced here: destructive verbs render a pending row
after first Enter ("Press Enter again to delete album"); second Enter
within 3s executes, Escape cancels.

### v1.5 — Selection verbs

`CommandContext` extends to expose `selectedAssetIds: string[]`. Palette
can be opened from the selection toolbar with selection pre-loaded.

Verbs (gated by non-empty selection):

- Add selected to album…
- Add selected to space…
- Favorite selected
- Archive selected
- Delete selected (destructive — inline confirm)

Multi-step commands: "Add to album…" switches palette into a secondary
state (title pinned, input re-scoped to album search, Escape returns to
main). This is a bigger UX change and probably its own 2-PR slice.
