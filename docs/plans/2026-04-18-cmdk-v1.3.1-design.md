# Design: cmdk palette v1.3.1 — Admin queue verbs

Direct extension of v1.3.0 (PR #374 on `feat/cmdk-next`). Adds 8 admin-only
`CommandItem`s that wrap existing queue-management primitives — run a
specific job, pause/resume all queues, clear failed jobs. No new
infrastructure; every verb is a single-call (or `Promise.allSettled`
fan-out) handler that drops into the existing `COMMAND_ITEMS` array.

The v1.3 main design doc sketched this sub-version as "Settings toggles &
admin globals." Reconnaissance showed the settings toggles (map-tile
theme, `showInTimeline`, auto-classification) don't have clean one-call
primitives — they require fetch-merge-put flows against
`updateMyPreferences` or `systemConfig`. Deferred to v1.3.2 / v1.4 pending
a small helper for that pattern. v1.3.1 ships the admin queue verbs only,
which share the `runQueueCommandLegacy` / `emptyQueue` primitive shape
with v1.3.0's commands.

## Non-goals (v1.3.1)

- **Settings toggles** (auto-classification on/off, map-tile-theme, user prefs) — deferred to v1.3.2 pending fetch-merge-put plumbing.
- **Confirmation UX** for "Clear failed jobs" (mildly destructive) — inline-confirm infra is v1.4 per v1.3 main design. v1.3.1 matches sign-out precedent (no confirm, accepted).
- **Cross-tab admin-panel refresh** when a palette-triggered job changes queue state — admin panel's existing socket/poll refresh covers it within seconds.
- **"Run job…" multi-step meta-command** that opens a sub-picker — multi-step infra is v1.5.
- **`force: true` variant** of `Start` for re-processing all assets — the palette defaults to `force: false`; admins who need force go to the admin panel.
- **Per-queue Pause / Resume / Clear** — palette ships bulk ("all queues") only. Admin panel keeps per-queue controls.
- **E2E Playwright coverage** — the v1.3.0 suite already covers the command-activation plumbing end-to-end. Adding 8 more cases for thin wrapper handlers triples the E2E runtime for zero new code-path coverage. Unit tests with SDK spies prove the wiring.

All `Start` commands dispatch with `force: false` (process missing/pending only). `Pause`/`Resume`/clear iterate the **admin-panel queue list** (see below) via `Promise.allSettled` so one queue failing never drops the rest.

### Shared queue list

`Object.values(QueueName)` contains 21 entries — including background/system queues that admins do not manage from the admin panel (`BackgroundTask`, `Search`, `Notification`, `BackupDatabase`, `Workflow`, `Editor`, `StorageBackendMigration`). Pausing those via the palette would silently stop things like notifications and background housekeeping, which a user typing "pause all queues" almost certainly does not intend.

Instead, v1.3.1 re-uses the 14 queues `QueuePanel.svelte` already renders. Today that list is implicit — it's the keys of a `queueDetails: Partial<Record<QueueName, QueueDetails>>` object literal inside the component, iterated via `Object.entries(queueDetails)`. The values contain `$t()` calls and `featureFlags` references, so the map itself can't move out of the component. But the _list of keys_ can.

**Prerequisite refactor** — make the list explicit and shared.

Append to `web/src/lib/constants.ts` (the existing shared constants module):

```ts
export const ADMIN_VISIBLE_QUEUES = [
  QueueName.ThumbnailGeneration,
  QueueName.MetadataExtraction,
  QueueName.Library,
  QueueName.Sidecar,
  QueueName.SmartSearch,
  QueueName.DuplicateDetection,
  QueueName.FaceDetection,
  QueueName.FacialRecognition,
  QueueName.Ocr,
  QueueName.PetDetection,
  QueueName.Classification,
  QueueName.VideoConversion,
  QueueName.StorageTemplateMigration,
  QueueName.Migration,
] as const satisfies readonly QueueName[];
```

`QueuePanel.svelte` imports `ADMIN_VISIBLE_QUEUES`, tightens `queueDetails`'s type to `Record<(typeof ADMIN_VISIBLE_QUEUES)[number], QueueDetails>` (drops the `Partial`), and iterates `{#each ADMIN_VISIBLE_QUEUES as name}` instead of `Object.entries(queueDetails)`. `command-items.ts` imports `ADMIN_VISIBLE_QUEUES` and uses it directly in the bulk handlers.

With the tightened type, TypeScript rejects the build if `queueDetails` is missing any name in `ADMIN_VISIBLE_QUEUES` or defines one outside it. Upstream adding a new queue to the admin panel forces an `ADMIN_VISIBLE_QUEUES` edit at compile time, and palette automatically picks it up through the shared import — no drift possible by construction, no runtime drift-guard test needed.

"All queues" in the palette means "all queues the admin panel shows" — the natural user model.

## Architecture

### Registry extension

Append 8 entries to `COMMAND_ITEMS` in `web/src/lib/managers/command-items.ts`. No type changes — `adminOnly?: boolean` already exists on `CommandItem` (reserved in v1.3.0 for exactly this).

### Handler shapes (three)

Helpers live alongside `COMMAND_ITEMS` inside `command-items.ts` so they stay with the registry they serve:

```ts
async function runQueue(name: QueueName) {
  const $t = get(t);
  const item = asQueueItem($t, { name });
  try {
    await runQueueCommandLegacy({
      name,
      queueCommandDto: { command: QueueCommand.Start, force: false },
    });
    toastManager.primary($t('cmdk_cmd_job_started', { values: { job: item.title } }));
  } catch (error) {
    handleError(error, $t('errors.something_went_wrong'));
  }
}

async function bulkQueueCommand(command: QueueCommand.Pause | QueueCommand.Resume) {
  const results = await Promise.allSettled(
    ADMIN_VISIBLE_QUEUES.map((name) => runQueueCommandLegacy({ name, queueCommandDto: { command } })),
  );
  const failed = results.filter((r) => r.status === 'rejected').length;
  const $t = get(t);
  if (failed > 0) {
    toastManager.warning($t('cmdk_cmd_bulk_partial', { values: { failed, total: results.length } }));
  } else {
    toastManager.primary($t(command === QueueCommand.Pause ? 'cmdk_cmd_all_paused' : 'cmdk_cmd_all_resumed'));
  }
}

async function clearAllFailedJobs() {
  const results = await Promise.allSettled(
    ADMIN_VISIBLE_QUEUES.map((name) => emptyQueue({ name, queueDeleteDto: { failed: true } })),
  );
  const failed = results.filter((r) => r.status === 'rejected').length;
  const $t = get(t);
  if (failed > 0) {
    toastManager.warning($t('cmdk_cmd_bulk_partial', { values: { failed, total: results.length } }));
  } else {
    toastManager.primary($t('cmdk_cmd_failed_cleared'));
  }
}
```

Each entry's `handler` is a thin arrow: `() => runQueue(QueueName.ThumbnailGeneration)`, etc. The toast's `{job}` value is resolved from `asQueueItem` (in `web/src/lib/services/queue.service.ts`), which maps each `QueueName` to its existing noun-form label (`admin.thumbnail_generation_job` = "Thumbnail generation", etc.). Result: toast reads "Started: Thumbnail generation" — avoids the double-verb "Started: Run thumbnail generation" we'd get if we passed the palette row's verb-form label through.

### Activation semantics (unchanged)

`activate('command')` wraps `cmd.handler()` in `queueMicrotask` with a `.catch((error) => console.error('[cmdk] command handler failed', …))` (v1.3.0 — see `global-search-manager.svelte.ts:1180-1184`). Our handlers `try/catch` + `handleError` internally, so the returned promise always resolves — the manager's `.catch` never fires for our handlers and only serves as a safety net for future commands that forget to catch. Exactly one toast appears per invocation (from `handleError` or the success path); no console noise in the success or caught-error case.

### State invalidation

Our direct API calls bypass `queue.service.ts`'s `eventManager.emit('QueueUpdate', ...)`. If the admin panel is open in another tab, it won't auto-refresh post-palette-trigger. **Accepted** — admin panel's existing socket/poll refresh catches up within seconds. Cross-tab sync is not worth the plumbing for this corner case.

## Gating

All 8 items carry `adminOnly: true`. `filterNavAndCommands` (v1.3.0) already drops items failing the admin check via `user.isAdmin`. The defensive gating test shipped in v1.3.0 (`global-search-manager.svelte.spec.ts`, "defensive gating: fabricated adminOnly command is filtered for non-admin user") covers this path. v1.3.1 adds **two new assertions** that pin real v1.3.1 items through the gate: a loop over all 8 v1.3.1 ids in the manager spec (end-to-end filter), plus a structural `adminOnly: true` check in the registry spec (catches a missing flag without needing the full manager pipeline).

## Error handling

| Failure mode                       | Behavior                                                                                                                                                                                                 |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Single-queue `Start` rejects       | `handleError(error, 'errors.something_went_wrong')` shows red toast; logged via `.catch`.                                                                                                                |
| One of N bulk calls rejects        | `Promise.allSettled` keeps the rest running; warning toast reports `{failed}/{total}` failed.                                                                                                            |
| All of N bulk calls reject         | Same warning toast; `failed === total`; admin should check queue state manually.                                                                                                                         |
| `runQueueCommandLegacy` 403/401    | `handleError` handles; UI redirects to login via existing auth-manager reactions if applicable.                                                                                                          |
| Admin rank revoked mid-flight      | `Promise.allSettled` is already in-flight — each running call gets a 403 individually and lands in the failed bucket; toast reports `{failed}/{total} failed`. Acceptable (rare, correct, discoverable). |
| Rapid double-Enter on same command | Covered by v1.3.0's `commandInFlight` single-flight guard (key = `cmd.id`). Second call no-ops until the first resolves.                                                                                 |
| Pausing an already-paused queue    | Server treats it as a no-op; our success toast "All queues paused" remains accurate post-condition. Same for Resume on an already-running queue.                                                         |

## Testing

### Unit (extend `command-items.spec.ts`)

#### SDK mock setup

Mock `@immich/sdk` at the top of the spec, matching the shape of the existing `@immich/ui` mock, so `runQueueCommandLegacy` and `emptyQueue` are `vi.fn()`s from the first test run. Tests can then use `vi.spyOn(sdk, 'runQueueCommandLegacy').mockResolvedValue(...)` (or `.mockRejectedValue(...)`) per-test, matching v1.3.0's `fileUploader` / `albumUtils` style.

```ts
vi.mock('@immich/sdk', async (orig) => ({
  ...(await orig<typeof import('@immich/sdk')>()),
  runQueueCommandLegacy: vi.fn(),
  emptyQueue: vi.fn(),
}));
```

Preserving `...(await orig(...))` keeps `QueueName` / `QueueCommand` enums intact — tests import and use those as real values. `import * as sdk from '@immich/sdk'` is the namespace handle used for `vi.spyOn` call sites.

#### Mock-state hygiene across parameterized suites

The `@immich/ui` mock's `toastManager.primary` / `.warning` / `.info` are persistent `vi.fn()` instances — calls accumulate across tests. The new `describe.each` suites assert both `toHaveBeenCalled()` and `not.toHaveBeenCalled()`, so cross-iteration leakage would produce false positives on the second and later iterations. Add a suite-level `beforeEach` that clears toast mocks and SDK spies:

```ts
beforeEach(() => {
  vi.mocked(toastManager.primary).mockClear();
  vi.mocked(toastManager.warning).mockClear();
  vi.mocked(toastManager.info).mockClear();
  vi.restoreAllMocks(); // resets vi.spyOn handles between iterations
});
```

#### Run-job commands (5 items) — parameterized table

```ts
describe.each([
  ['cmd:run_thumbnail_gen', QueueName.ThumbnailGeneration],
  ['cmd:run_metadata_extraction', QueueName.MetadataExtraction],
  ['cmd:run_smart_search', QueueName.SmartSearch],
  ['cmd:run_face_detection', QueueName.FaceDetection],
  ['cmd:run_face_recognition', QueueName.FacialRecognition],
])('%s', (id, expectedQueue) => {
  it('dispatches Start + force:false against the expected queue', async () => {
    const spy = vi.spyOn(sdk, 'runQueueCommandLegacy').mockResolvedValue({} as never);
    const cmd = COMMAND_ITEMS.find((c) => c.id === id)!;
    await cmd.handler();
    expect(spy).toHaveBeenCalledWith({
      name: expectedQueue,
      queueCommandDto: { command: QueueCommand.Start, force: false },
    });
    // Pin toast CONTENT, not just calledness — catches a copy-paste that fires the
    // wrong i18n key (e.g. showing the paused-queues toast on a Start command).
    expect(toastManager.primary).toHaveBeenCalledWith(expect.stringContaining('cmdk_cmd_job_started'));
  });
});

// Shared helper — one rejection test covers the error branch for all 5 Run-X
// commands since they dispatch through `runQueue`.
it('cmd:run_thumbnail_gen: SDK rejection fires handleError (danger toast), no success toast', async () => {
  vi.spyOn(sdk, 'runQueueCommandLegacy').mockRejectedValue(new Error('boom') as never);
  const cmd = COMMAND_ITEMS.find((c) => c.id === 'cmd:run_thumbnail_gen')!;
  await cmd.handler();
  expect(toastManager.primary).not.toHaveBeenCalled();
  // handleError calls toastManager.danger (web/src/lib/utils/handle-error.ts:44).
  expect(toastManager.danger).toHaveBeenCalled();
});
```

Toast assertions use `stringContaining(<i18n key>)` throughout. v1.3.0's spec's `$t` loader returns the key itself when no catalog is loaded, so the rendered string IS the key — an ideal fingerprint for which path fired without coupling tests to translated copy.

The parameterized table pins `force: false` for **every** Start command — a future copy-paste flipping one to `true` breaks the row for that id, not just one hand-written case.

#### Bulk pause / resume (2 items)

```ts
describe.each([
  ['cmd:pause_all_queues', QueueCommand.Pause],
  ['cmd:resume_all_queues', QueueCommand.Resume],
])('%s', (id, expectedCommand) => {
  it('dispatches to every palette queue (not all QueueName values)', async () => {
    const spy = vi.spyOn(sdk, 'runQueueCommandLegacy').mockResolvedValue({} as never);
    const cmd = COMMAND_ITEMS.find((c) => c.id === id)!;
    await cmd.handler();
    expect(spy).toHaveBeenCalledTimes(ADMIN_VISIBLE_QUEUES.length);
    for (const name of ADMIN_VISIBLE_QUEUES) {
      expect(spy).toHaveBeenCalledWith({ name, queueCommandDto: { command: expectedCommand } });
    }
    // Explicit negative: a system queue excluded from ADMIN_VISIBLE_QUEUES must NOT be touched.
    expect(spy).not.toHaveBeenCalledWith(expect.objectContaining({ name: QueueName.Notification }));
    // Toast content: Pause → cmdk_cmd_all_paused, Resume → cmdk_cmd_all_resumed.
    // Catches a ternary inversion in bulkQueueCommand.
    const expectedToastKey = expectedCommand === QueueCommand.Pause ? 'cmdk_cmd_all_paused' : 'cmdk_cmd_all_resumed';
    expect(toastManager.primary).toHaveBeenCalledWith(expect.stringContaining(expectedToastKey));
  });

  it('partial failure: some reject → warning toast fires, no success toast', async () => {
    const spy = vi
      .spyOn(sdk, 'runQueueCommandLegacy')
      .mockImplementation(({ name }) =>
        name === QueueName.ThumbnailGeneration
          ? (Promise.reject(new Error('boom')) as never)
          : (Promise.resolve({}) as never),
      );
    const cmd = COMMAND_ITEMS.find((c) => c.id === id)!;
    await cmd.handler();
    expect(spy).toHaveBeenCalledTimes(ADMIN_VISIBLE_QUEUES.length);
    // Toast assertion is key-only: svelte-i18n with no catalog loaded returns the
    // raw key AND skips ICU substitution, so interpolated {failed}/{total} values
    // can't be asserted. The reject-selection above proves the 1/14 breakdown.
    expect(toastManager.warning).toHaveBeenCalledWith(expect.stringContaining('cmdk_cmd_bulk_partial'));
    expect(toastManager.primary).not.toHaveBeenCalled();
  });

  it('total failure: all reject → warning toast fires, no success toast', async () => {
    const spy = vi.spyOn(sdk, 'runQueueCommandLegacy').mockRejectedValue(new Error('boom') as never);
    const cmd = COMMAND_ITEMS.find((c) => c.id === id)!;
    await cmd.handler();
    expect(spy).toHaveBeenCalledTimes(ADMIN_VISIBLE_QUEUES.length);
    expect(toastManager.warning).toHaveBeenCalledWith(expect.stringContaining('cmdk_cmd_bulk_partial'));
    expect(toastManager.primary).not.toHaveBeenCalled();
  });
});
```

#### Registry & drift guards

- **Registry length**: `command-items.spec.ts` today has no `COMMAND_ITEMS.length` assertion — only a duplicate-id check. Add `expect(COMMAND_ITEMS).toHaveLength(15)` (7 v1.3.0 + 8 v1.3.1) so a future PR can't silently drop an entry.
- **All v1.3.1 entries are `adminOnly`**: loop over the 8 v1.3.1 ids asserting each has `adminOnly: true` directly on its `COMMAND_ITEMS` entry. Cheap structural check — catches a missing flag on any single new command without spinning up the manager pipeline.
- **Queue-list drift**: no runtime test. The prerequisite refactor's tightened `Record<(typeof ADMIN_VISIBLE_QUEUES)[number], QueueDetails>` type on `queueDetails` makes drift structurally impossible — TypeScript flags the build if the two diverge.

### Unit (extend `global-search-manager.svelte.spec.ts`)

- **Admin gating across all v1.3.1 items**: one test that loops over the 8 v1.3.1 ids, asserting each is absent from `sections.commands` for a non-admin user and present for an admin. Identical shape to v1.3.0's fabricated-admin test, but targeting the live registry — catches a missing `adminOnly` on any single new command via the real filter pipeline.
- **`commandInFlight` guard covers v1.3.1 commands**: press Enter twice on `cmd:run_thumbnail_gen` while the first handler's promise is still pending; assert `runQueueCommandLegacy` was called exactly once. (The guard is v1.3.0 code, but the test pins that it still applies to v1.3.1 handlers with their added `try/catch` layer.) **Setup note**: no `@immich/sdk` mock is needed at the manager-spec layer — stub `command-items.ts` locally or replace the handler in the test with a function returning a never-resolving `Promise`. Pulling in the full SDK mock just for the guard test is overkill.

## Alternatives considered

- **Grouped meta-commands** ("Run media jobs", "Run ML jobs", "Manage queues") — rejected. Reduces row count from 8 to 3 but loses specificity; users can't tell what a group does without docs.
- **Multi-step `cmd:run_job…` with sub-picker** — rejected. Multi-step command infra is v1.5; pulling it forward balloons v1.3.1's scope.
- **Use existing `handlePauseQueue(queueResponseDto)` from `queue.service.ts`** — rejected for bulk. That helper takes a full `QueueResponseDto`, which requires a per-queue GET first. Direct `runQueueCommandLegacy({name, ...})` in a fan-out is simpler and cheaper. Single-queue `Run X` commands also use direct dispatch for consistency.
- **Cross-tab `eventManager.emit('QueueUpdate')` after palette dispatch** — rejected. Admin panel's socket/poll covers it within seconds; not worth the plumbing.
- **Per-queue Pause / Resume rows** — rejected. That's 3 commands × N queues = registry bloat. Bulk-all is the palette use case; admin panel has the fine-grained controls.
